# End-to-End Evaluation Design #001

Status: Evaluation Design Draft, No Runtime Changes
Owner: LYNCA Listing Intelligence
Generated: 2026-06-22

## Scope

This document defines how to measure real Listing Copilot accuracy as a complete system.

Evaluation flow:

```text
Image-backed feedback record
↓
Original images
↓
Current Vision
↓
Current Prompt
↓
Generated Title
↓
Compare against corrected_title
```

No runtime code, registry, resolver, prompt, deployment, tests, or upgrades are modified.

## Core Answer

Measure Listing Copilot accuracy by field-level agreement, not exact title-string match.

Exact title matching is too brittle because operators may change ordering, wording, abbreviations, or style. The useful measure is whether the generated title preserves the card facts that matter:

- product
- set
- serial
- auto
- subject
- parallel

The corrected title should be treated as the operator-approved target, but not as unquestioned truth for checklist-dependent fields. Ambiguous cases should be marked for human review.

## Evaluation Unit

Each evaluation row should represent one image-backed feedback record.

Required inputs:

| Field | Purpose |
| --- | --- |
| `feedback_id` | Stable row identifier. |
| `front_image_url` | Front image evidence. |
| `back_image_url` | Back image evidence when available. |
| `corrected_title` | Operator-approved target title. |
| `current_generated_title` | Title generated by the current full pipeline. |
| `vision_output` | Structured or text output from Current Vision. |
| `prompt_output` | Final generated title after Current Prompt. |

Recommended metadata:

| Field | Purpose |
| --- | --- |
| `operator_id` | Trace review source. |
| `created_at` | Track drift over time. |
| `image_backed_status` | `front_only`, `front_and_back`, or `missing`. |
| `requires_checklist` | Human-review flag for SSP, case-hit, exact insert, set-year, or relic naming. |
| `review_notes` | Human adjudication notes. |

## Field Extraction

For each row, extract comparable fields from both generated and corrected titles.

Target schema:

```json
{
  "product": "",
  "set": "",
  "serial": "",
  "auto": "",
  "subject": [],
  "parallel": ""
}
```

Extraction sources:

| Source | Use |
| --- | --- |
| `corrected_title` | Target field extraction. |
| generated title | Candidate field extraction. |
| image evidence | Human or vision adjudication for visible fields. |
| checklist source | Required only for checklist-dependent fields. |

Important:

Do not count a field as wrong only because wording differs. Count it wrong when the underlying card fact differs, is missing, or is unsupported.

## Metrics

### Product Accuracy

Definition:

The generated title preserves the correct manufacturer/product family, such as `Topps`, `Panini`, `Bowman`, `Upper Deck`, `Leaf`, or product family text present in the target.

Scoring:

| Score | Meaning |
| --- | --- |
| `pass` | Product/manufacturer matches target or is an accepted equivalent. |
| `fail_missing` | Product is absent from generated title but present in target. |
| `fail_wrong` | Product is wrong. |
| `not_applicable` | Target has no product/manufacturer field. |
| `needs_review` | Product identity depends on unclear image/checklist evidence. |

Primary metric:

```text
product_accuracy = product_pass / product_applicable
```

### Set Accuracy

Definition:

The generated title preserves the correct set, product line, subset, or edition when target evidence supports it.

Examples:

- `Topps Chrome`
- `Cosmic Chrome`
- `Bowman Chrome`
- `Sapphire Edition`
- `Donruss Optic`
- `Topps Chrome Tennis`

Scoring:

| Score | Meaning |
| --- | --- |
| `pass` | Set/product-line matches target or accepted equivalent. |
| `fail_missing` | Specific set is missing or collapsed to generic product. |
| `fail_wrong` | Set/product-line is wrong. |
| `not_applicable` | Target has no set/product-line field. |
| `needs_review` | Set identity requires checklist or ambiguous visual evidence. |

Primary metric:

```text
set_accuracy = set_pass / set_applicable
```

### Serial Accuracy

Definition:

The generated title preserves the exact serial number when target includes one.

Examples:

- `1/1`
- `31/50`
- `033/100`
- `29/199`

Scoring:

| Score | Meaning |
| --- | --- |
| `pass` | Exact serial matches target. |
| `fail_missing` | Target serial is absent from generated title. |
| `fail_wrong` | Numerator or denominator differs. |
| `fail_duplicate` | Serial appears duplicated or conflicting. |
| `not_applicable` | Target has no serial. |
| `needs_review` | Serial is unreadable or target appears uncertain. |

Primary metric:

```text
serial_accuracy = serial_pass / serial_applicable
```

### Auto Accuracy

Definition:

The generated title preserves autograph status and autograph-specific grade semantics.

Scoring:

| Score | Meaning |
| --- | --- |
| `pass` | Auto/autograph/signature status matches target. |
| `fail_missing` | Target indicates auto but generated title omits it. |
| `fail_wrong` | Generated title says auto when target does not, or uses misleading auto wording. |
| `fail_grade_split` | Card grade and auto grade are merged, dropped, or misrepresented. |
| `not_applicable` | Target has no auto field. |
| `needs_review` | Auto status depends on unclear image/slab evidence. |

Primary metric:

```text
auto_accuracy = auto_pass / auto_applicable
```

### Subject Accuracy

Definition:

The generated title preserves all named card subjects from the target title.

Subjects may include:

- players
- athletes
- characters
- artists
- dual/triple/quad signers

Scoring:

| Score | Meaning |
| --- | --- |
| `pass` | All target subjects are present. |
| `fail_missing_subject` | One or more target subjects are missing. |
| `fail_wrong_subject` | Wrong subject appears. |
| `fail_order_only` | All subjects present but order differs; normally not a factual failure. |
| `not_applicable` | Target has no identifiable subject. |
| `needs_review` | Subject identity is ambiguous. |

Primary metric:

```text
subject_accuracy = subject_pass / subject_applicable
```

### Parallel Accuracy

Definition:

The generated title preserves the correct parallel or variant only when target evidence supports that field.

Examples:

- `Gold Refractor`
- `Raywave Refractor`
- `Geometric`
- `Shimmer`
- `Sapphire`
- `Padparadscha`

Scoring:

| Score | Meaning |
| --- | --- |
| `pass` | Parallel matches target or accepted equivalent. |
| `fail_missing` | Target parallel is missing. |
| `fail_wrong` | Generated parallel differs from target. |
| `fail_overclaim` | Generated title adds unsupported parallel. |
| `not_applicable` | Target has no parallel. |
| `needs_review` | Parallel requires visual/checklist adjudication. |

Primary metric:

```text
parallel_accuracy = parallel_pass / parallel_applicable
```

## Aggregate Metrics

Report both field-level and record-level metrics.

Field-level:

| Metric | Meaning |
| --- | --- |
| `product_accuracy` | Product pass rate. |
| `set_accuracy` | Set pass rate. |
| `serial_accuracy` | Serial pass rate. |
| `auto_accuracy` | Auto pass rate. |
| `subject_accuracy` | Subject pass rate. |
| `parallel_accuracy` | Parallel pass rate. |

Record-level:

| Metric | Meaning |
| --- | --- |
| `all_field_accuracy` | Record passes every applicable field. |
| `critical_field_accuracy` | Product, subject, serial, auto, and parallel pass when applicable. |
| `high_value_error_rate` | Any serial, auto-grade, wrong subject, unsupported parallel, or unsupported scarcity error. |
| `needs_review_rate` | Share of records requiring human/checklist adjudication. |
| `overclaim_rate` | Share of records where generated title adds unsupported fields. |
| `omission_rate` | Share of records where generated title drops target fields. |

## Error Taxonomy

Every failed field should receive one error type:

| Error type | Meaning |
| --- | --- |
| `missing_field` | Target has a field that generated title omitted. |
| `wrong_field` | Generated title has incorrect field value. |
| `overclaim` | Generated title adds unsupported field. |
| `duplicate_field` | Generated title repeats field confusingly. |
| `format_only` | Field is correct but formatting differs. |
| `needs_checklist` | Cannot adjudicate without external source. |
| `needs_image_review` | Cannot adjudicate without image inspection. |

## Human Review Rules

Human review is required when:

- corrected title includes `SSP`, `SP`, or case-hit language
- corrected title includes exact checklist insert names
- corrected title includes `Sapphire`, `Shimmer`, `Raywave`, `Geometric`, or other visual parallel not clearly represented in text
- serial number is unreadable or conflicting
- subject identity is ambiguous
- generated title and corrected title disagree but both appear plausible

Human review should record:

- whether the corrected title is accepted as target truth
- whether the field is visible, text-supported, checklist-supported, or unsupported
- whether the row should be excluded from automated metric calculation

## A/B Evaluation Design

Use this design for every proposed system change.

Comparison:

| Run | Description |
| --- | --- |
| Baseline | Current Vision + Current Prompt. |
| Candidate | Proposed Vision/Prompt/Registry/Resolver change. |

For each run:

1. Process the same image-backed records.
2. Generate titles without seeing `corrected_title`.
3. Extract structured fields from generated titles.
4. Extract structured target fields from corrected titles.
5. Score field-level accuracy.
6. Compare baseline versus candidate.

A change should not ship unless:

- target metrics improve,
- high-value error rate does not increase,
- overclaim rate does not increase,
- needs-review cases are not silently converted into confident claims,
- human review accepts the change.

## Suggested First Evaluation

Run a 100-record standard evaluation using stratified sampling:

| Stratum | Suggested count |
| --- | ---: |
| Parallel-heavy corrections | 25 |
| Product/set corrections | 25 |
| Serial corrections | 15 |
| Auto/grade corrections | 15 |
| Subject/team corrections | 10 |

Also run a 25-record smoke evaluation before any full run.

## Reporting Template

Each evaluation report should include:

| Section | Contents |
| --- | --- |
| Dataset | Record count, image-backed count, date range. |
| System version | Vision version, prompt version, registry/resolver version. |
| Field metrics | Product, set, serial, auto, subject, parallel accuracy. |
| Record metrics | All-field accuracy, critical-field accuracy, overclaim rate, omission rate. |
| Error breakdown | Counts by error type. |
| Human-review queue | Ambiguous/checklist-dependent records. |
| Top regressions | Worst new failures versus baseline. |
| Recommendation | Ship, revise, reject, or more review. |

## What Not To Measure As Primary Accuracy

Do not use exact string match as the primary metric.

Do not reward generated titles for adding unsupported high-value terms.

Do not treat `corrected_title` as final truth for checklist-dependent claims without review.

Do not hide ambiguous rows by forcing pass/fail when `needs_review` is the correct outcome.

## Final Answer

Real Listing Copilot accuracy should be measured by field-level factual agreement between the generated title and the corrected title, adjudicated against image evidence and checklist review where needed.

The core scorecard should be:

```text
product_accuracy
set_accuracy
serial_accuracy
auto_accuracy
subject_accuracy
parallel_accuracy
critical_field_accuracy
high_value_error_rate
overclaim_rate
needs_review_rate
```

The system is accurate when it preserves the correct card facts without adding unsupported high-value claims.

## Non-Goals

This design does not:

- modify runtime code
- modify prompts
- modify registry data
- modify resolver logic
- install upgrades
- create automated tests
- download images
