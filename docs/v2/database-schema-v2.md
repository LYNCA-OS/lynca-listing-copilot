# Listing Copilot Database Schema v2

Status: Design Draft v2.0
Owner: LYNCA Listing Intelligence
Companion Documents:

- `feedback-loop-v2.md`
- `ui-wireframe-v2.md`

## Purpose

This document defines the smallest database design needed for Listing Copilot V2 feedback memory with image evidence.

It is a schema design document only. It does not provide implementation code or migration SQL.

## Design Principle

Store the correction and the image evidence. Do not ask the operator to explain it.

Required memory:

- generated title
- corrected title
- front image URL or front image reference
- back image URL or back image reference
- timestamp
- operator

The V2.0 memory unit is an image-title pair record. Text-only feedback is not sufficient for V2.0 because future registry review and visual distinction training require the image evidence that produced the correction.

## Table Overview

V2.0 needs one new core table:

```text
listing_title_feedback
```

This table stores title correction events created when an operator saves a listing title that differs from the generated title.

Each memory unit includes the title pair and the image evidence that produced the correction.

## Table: listing_title_feedback

### Purpose

Stores raw before/after title feedback with front/back image evidence.

Each row represents one saved correction event.

### Fields

| Field | Required | Purpose |
| --- | --- | --- |
| id | Yes | Unique feedback event identifier |
| generated_title | Yes | Title originally generated by Listing Copilot |
| corrected_title | Yes | Final title saved by the operator |
| front_image_url | Conditional | URL for front image evidence if URL storage is used |
| front_image_reference | Conditional | Internal reference for front image evidence if reference storage is used |
| back_image_url | Conditional | URL for back image evidence if URL storage is used and a back image exists |
| back_image_reference | Conditional | Internal reference for back image evidence if reference storage is used and a back image exists |
| created_at | Yes | Timestamp when the correction was saved |
| operator_id | Yes | Operator who saved the corrected title |
| listing_session_id | Optional | Links feedback to a generation session if one exists |
| source_listing_id | Optional | Links feedback to a saved listing if one exists |
| image_batch_id | Optional | Links feedback to uploaded image set if one exists |

### Image Field Rule

V2.0 must store either image URLs or image references.

At minimum:

- one front image URL or front image reference is required
- one back image URL or back image reference is required when the asset has a back image

The schema should not require both URL and reference forms for the same image. The project should choose the storage pattern that best matches the implementation environment.

### Required Field Notes

`generated_title`

The system-generated title before operator edits.

`corrected_title`

The final saved title after operator edits.

`front_image_url` or `front_image_reference`

The front image evidence associated with the generated and corrected title pair.

`back_image_url` or `back_image_reference`

The back image evidence associated with the generated and corrected title pair when a back image exists.

`created_at`

The timestamp of the save event.

`operator_id`

The authenticated operator responsible for the save action.

### Optional Field Notes

Optional fields should only be included when the surrounding app already has the data.

V2.0 should not add new mandatory workflow steps to populate optional metadata.

## Excluded Fields

The V2.0 feedback table should not include required fields for:

- error category
- tags
- explanation
- classification
- severity
- confidence
- registry target
- correction type
- model version

Some of these may become useful later, but making them part of V2.0 risks turning a low-friction memory system into a labeling tool.

## Event Creation Rule

A feedback row is created when:

```text
corrected_title differs from generated_title
```

The feedback row must include image evidence for the same asset.

Recommended V2.0 behavior:

- If the title changed, store a feedback row.
- If the title did not change, do not store a row.

Accepted-title tracking can be reconsidered in V2.1.

## Supabase Architecture

### Components

V2.0 Supabase responsibilities:

- authentication identity for `operator_id`
- feedback event storage
- image evidence storage or image evidence references
- admin read access
- future export or analytics base

Application responsibilities:

- preserve the generated title during the edit session
- preserve the front/back image evidence reference during the edit session
- compare generated and saved title
- write a feedback event after Save
- keep normal listing save behavior intact

### Data Flow

```text
Operator Uploads Images
  |
Listing Copilot Generates Title
  |
Frontend Keeps generated_title
  |
Operator Edits Title
  |
Operator Saves
  |
Application Compares Titles
  |
Supabase Stores Feedback Row With Image Evidence
```

### Access Model

Operator access:

- can create feedback rows for their own save actions
- does not need a separate feedback-management screen

Admin access:

- can read feedback rows
- can filter or export in later versions
- should not need to modify raw feedback in V2.0

### Data Ownership

Feedback rows are historical records.

Raw feedback should remain immutable in V2.0. If future review state is needed, it should be added as separate review metadata rather than rewriting the original generated and corrected titles.

Image references should also remain immutable. If an image is reprocessed later, it should create new derived metadata rather than changing the original evidence pointer.

## Future Tables

V2.0 should avoid adding these tables immediately:

- feedback_tags
- feedback_categories
- correction_labels
- registry_review_queue
- model_training_examples

V2.0 should also avoid adding visual-label tables. Image evidence is stored so future reviewers can inspect distinctions later without asking operators to classify them during Save.

These may be introduced in V2.1 or later if review volume proves high enough.

## V2.1 Schema Expansion

Potential V2.1 additions:

| Addition | Purpose |
| --- | --- |
| accepted_title_events | Track unchanged generated titles |
| reviewed_at | Mark admin-reviewed feedback |
| reviewed_by | Identify reviewing admin |
| registry_candidate_id | Link a feedback row to a registry candidate |
| generated_title_hash | Group repeated generated titles |
| corrected_title_hash | Group repeated corrected titles |
| visual_distinction_notes | Admin-only notes for future visual review |

V2.1 additions should remain optional for normal operators.

## Non-Goals

This schema does not define:

- retraining pipeline tables
- vector memory
- prompt versioning
- model evaluation datasets
- automated registry writes
- detailed correction taxonomy

The V2.0 goal is durable memory, not automated learning.
