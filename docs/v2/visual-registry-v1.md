## Purpose

The Visual Registry V1 is a human-reviewed structure for image-backed visual concepts.

It stores visual concepts that repeatedly appear in feedback review, such as:

- Gold Wave
- Sapphire
- Star Fractor
- Geometric
- Padparadscha
- Cosmic Chrome

The registry is not an automatic learning system. It is a review artifact that helps admins collect visual evidence, compare recurring confusions, and decide whether a concept is ready to become a registry proposal, resolver proposal, prompt proposal, or regression test case.

## Core Principle

Visual evidence comes before title interpretation.

Each Visual Registry entity should answer:

- What does this concept look like?
- Which image examples support it?
- What does Listing Copilot commonly confuse it with?
- How much feedback evidence exists?
- Has a human reviewed the evidence?
- Which feedback records and upgrade proposals are linked?

## Non-Goals

Visual Registry V1 does not:

- modify runtime title generation
- mutate the existing title registry
- add resolver behavior
- edit prompts
- auto-approve visual concepts
- train or fine-tune a model
- create RAG or vector memory
- require operator labels
- replace admin review

## Entity Model

Each visual concept is represented as one entity.

Required fields:

| Field | Purpose |
| --- | --- |
| `id` | Stable internal identifier for the visual concept |
| `canonical_name` | Human-approved name, such as `Gold Wave` |
| `representative_images` | Curated image evidence examples |
| `common_confusions` | Concepts or phrases commonly confused with this entity |
| `evidence_count` | Number of linked feedback records supporting the entity |
| `review_status` | Current human review state |
| `linked_feedback_ids` | Feedback rows supporting or questioning the concept |
| `linked_upgrade_proposals` | Upgrade proposals that reference this concept |

Recommended optional fields:

| Field | Purpose |
| --- | --- |
| `concept_family` | Grouping such as parallel, product finish, release design, auto/relic distinction |
| `description` | Short visual description written by an admin |
| `positive_indicators` | Visual cues that support the concept |
| `negative_indicators` | Visual cues that should prevent false positives |
| `title_terms` | Common title words associated with the concept |
| `do_not_confuse_with` | Stronger warning list for high-risk neighboring concepts |
| `first_seen_at` | Earliest linked feedback timestamp |
| `last_seen_at` | Latest linked feedback timestamp |
| `reviewed_by` | Admin who reviewed the entity |
| `reviewed_at` | Timestamp of last human review |
| `notes` | Admin-only review notes |

## Review Status

Recommended review statuses:

| Status | Meaning |
| --- | --- |
| `candidate` | Concept was surfaced by feedback review but has not been validated |
| `needs_more_evidence` | Concept may be real but needs more image examples or checklist confirmation |
| `approved_reference` | Concept has enough evidence to serve as a visual reference |
| `accepted_as_test_case` | Concept should become regression evidence but not a runtime rule |
| `proposal_ready` | Concept is ready to be linked to a concrete upgrade proposal |
| `rejected` | Concept should not be used as system knowledge |
| `deprecated` | Concept was once useful but should no longer be used |

`approved_reference` does not mean the runtime system can use the concept automatically. It only means admins trust the concept as a reviewed visual reference.

## Representative Images

Representative images should be curated, not merely counted.

Each representative image should include:

| Field | Purpose |
| --- | --- |
| `feedback_id` | Source feedback row |
| `front_image_url` | Front evidence URL |
| `back_image_url` | Back evidence URL, when available |
| `generated_title` | Title generated by Listing Copilot |
| `corrected_title` | Title saved by operator |
| `evidence_role` | `positive`, `negative`, or `confusion` |
| `review_note` | Admin note explaining why this image matters |

Evidence roles:

- `positive`: image supports the canonical concept
- `negative`: image looks similar but should not be labeled as the concept
- `confusion`: image demonstrates a common mistake or neighboring concept

Representative images should be limited enough for review. V1 should prefer 3-10 curated examples over dumping every linked feedback row into the visual entity.

## Common Confusions

Common confusions identify where the system or operator corrections show repeated ambiguity.

Each confusion should include:

| Field | Purpose |
| --- | --- |
| `confused_with` | Neighboring concept or phrase |
| `confusion_type` | Product, parallel, insert, auto/relic/patch, wording, or unknown |
| `evidence_count` | Number of linked examples showing this confusion |
| `example_feedback_ids` | Representative feedback ids |
| `risk_level` | Low, medium, or high |
| `review_note` | Admin explanation |

Example confusions:

- Gold Wave vs Gold Refractor
- Sapphire vs Chrome
- Cosmic Chrome vs Chrome
- Star Fractor vs Orange Refractor
- Geometric vs standard color Refractor
- Patch vs Relic
- Auto vs Non-Auto

## Evidence Count

`evidence_count` is a signal, not approval.

Recommended counting rule:

```text
evidence_count = unique linked feedback ids with image evidence reviewed for this concept
```

Evidence should be separated by role:

| Count | Meaning |
| --- | --- |
| `positive_evidence_count` | Feedback records supporting the concept |
| `confusion_evidence_count` | Feedback records showing confusion with another concept |
| `negative_evidence_count` | Feedback records reviewed as lookalikes or non-examples |

High evidence can move a concept toward review, but only human approval can move it toward proposals.

## Linked Feedback IDs

`linked_feedback_ids` connect the visual concept back to raw evidence.

Each linked feedback id should remain traceable to:

- front image URL
- back image URL
- generated title
- corrected title
- operator id
- created timestamp

Raw feedback should remain immutable. Visual Registry entities should reference raw feedback; they should not rewrite it.

## Linked Upgrade Proposals

`linked_upgrade_proposals` connect visual concepts to future system changes.

A visual concept may link to:

- registry candidate proposal
- resolver candidate proposal
- prompt candidate proposal
- test case candidate proposal
- documentation proposal

Linking a proposal does not install it. Installation remains a separate approval and implementation workflow.

## Seed Concepts For V1

The first Visual Registry review set should focus on concepts surfaced by Review Cycle #001.

### Gold Wave

Concept family:

```text
parallel
```

Common confusions:

- Gold Refractor
- Gold Shimmer
- Gold Geometric

Review focus:

- wave texture must be visible
- serial and back text should support the title when possible
- do not treat gold color alone as evidence

### Sapphire

Concept family:

```text
product finish / product line
```

Common confusions:

- Chrome
- Bowman Chrome
- Sapphire Edition
- standard color refractor

Review focus:

- distinguish product line from surface finish
- confirm whether `Sapphire` belongs in product, parallel, or insert/title term for the specific release

### Star Fractor

Concept family:

```text
parallel / visual finish
```

Common confusions:

- Orange Refractor
- Cosmic Chrome
- general SSP wording

Review focus:

- require visible star-like pattern or explicit back/checklist evidence
- avoid applying from color alone

### Geometric

Concept family:

```text
parallel / visual finish
```

Common confusions:

- standard color Refractor
- Mojo
- Wave
- Shimmer

Review focus:

- identify geometric surface pattern
- collect negative examples where color is correct but pattern is not

### Padparadscha

Concept family:

```text
parallel / rare color naming
```

Common confusions:

- Orange Refractor
- Pink Refractor
- Sapphire 1/1
- Superfractor-style assumptions

Review focus:

- high-risk name; require strong checklist or visible evidence
- likely best accepted as test case before resolver rule

### Cosmic Chrome

Concept family:

```text
product line / release design
```

Common confusions:

- Topps Chrome
- standard Chrome
- Star Fractor
- Cosmic-style visual background

Review focus:

- distinguish release/product identity from visual pattern
- avoid rewriting standard Chrome to Cosmic Chrome from visual style alone

## Relationship To Existing Systems

### Feedback Rows

Feedback rows are raw evidence.

Visual Registry entities reference feedback rows but do not change them.

### Learning Review Candidates

Learning review candidates surface repeated patterns.

Visual Registry entities curate the subset of those patterns that require image-backed concept review.

### Existing Registry

The existing title registry stores title knowledge.

The Visual Registry stores image-backed visual references. It should not automatically write to the title registry.

### Resolver Rules

Resolver rules may later use approved visual concepts as design references, but V1 does not create resolver behavior.

### Test Cases

Visual Registry entities are strong sources for regression test cases.

An entity can be accepted as a test case even when it is not ready for registry or resolver changes.

## Review Workflow

Recommended V1 workflow:

```text
Review Cycle Candidate
  |
Identify Visual Concept
  |
Create Or Update Visual Registry Entity
  |
Attach Representative Images
  |
Record Common Confusions
  |
Set Review Status
  |
Link Future Upgrade Proposals
```

Review rules:

- Always inspect images before changing status.
- Prefer narrow concepts over broad visual buckets.
- Keep positive, negative, and confusion examples separate.
- Treat subtle visual parallels as high risk by default.
- Use `accepted_as_test_case` when the example is useful but not enough for a rule.
- Do not install runtime behavior from Visual Registry status alone.

## Future Proposal Format

When a Visual Registry concept becomes proposal-ready, the proposal should lead with the evidence package:

```text
Evidence Package:
- representative image URLs
- generated titles
- corrected titles
- feedback ids

Visual Concept:
- canonical name
- common confusions
- evidence count
- review status

Proposed Change:

Risk:

Affected Files:

Recommended Tests:
```

## V1 Success Criteria

Visual Registry V1 is successful if:

- admins can see canonical visual concepts and their evidence
- recurring visual confusions are tracked explicitly
- each concept links back to feedback ids and image URLs
- image-specific corrections can become test cases without becoming runtime rules
- registry, resolver, prompt, and test proposals remain human-approved
- no automatic learning or mutation occurs
