## Purpose

This runbook explains how to turn an offline Supabase feedback export into admin review candidates.

The review pipeline is intentionally offline. It does not change Listing Copilot runtime behavior and does not write to the registry, resolver, prompt files, tests, or raw feedback data.

## Safety Rules

- Do not mutate raw feedback.
- Do not auto-approve candidates.
- Do not write to the registry from this script.
- Do not write to resolver logic from this script.
- Do not write to prompts from this script.
- Do not create RAG indexes from review candidates.
- Do not fine-tune models from review candidates.
- Do not require operator labels.
- Keep the process offline and admin-reviewed.
- Preserve the One Extra Click operator workflow.

## 1. Export Supabase Feedback

Export rows from the Supabase feedback table used by V2.0B.

The export must include these fields:

| Field | Required | Notes |
| --- | --- | --- |
| `id` | Yes | Stable feedback event id |
| `generated_title` | Yes | Title generated by Listing Copilot |
| `corrected_title` | Yes | Title saved by the operator |
| `front_image_url` | Yes | Stable front image evidence URL |
| `back_image_url` | No | Stable back image evidence URL, if present |
| `operator_id` | Yes | Operator who saved the correction |
| `created_at` | Yes | Save timestamp |

CSV and JSON exports are supported.

Recommended Supabase export filters:

- Export a bounded date range, such as one day or one week.
- Include only rows where `generated_title` and `corrected_title` are populated.
- Keep image URL fields even when a row has no back image.
- Do not edit exported title text before processing.

## 2. Place The Exported File

Place the exported file somewhere local in the repository or on the same machine.

Recommended location:

```text
data/learning/supabase-feedback-export-YYYY-MM-DD.csv
```

or:

```text
data/learning/supabase-feedback-export-YYYY-MM-DD.json
```

The export file is raw evidence. Treat it as internal data. Do not manually rewrite titles, image URLs, operator ids, or timestamps.

## 3. Run The Script

From the repository root:

```bash
node scripts/v2-learning-review.mjs data/learning/supabase-feedback-export-YYYY-MM-DD.csv
```

Optional output date override:

```bash
node scripts/v2-learning-review.mjs data/learning/supabase-feedback-export-YYYY-MM-DD.csv --date YYYY-MM-DD
```

Optional output directory override:

```bash
node scripts/v2-learning-review.mjs path/to/export.json --out-dir data/learning
```

The script writes:

```text
data/learning/review-candidates-YYYY-MM-DD.json
data/learning/review-candidates-YYYY-MM-DD.md
```

## 4. Interpret Review Candidates

Each candidate is a possible learning pattern. It is not an approved system upgrade.

Review candidates should be read evidence-first. Inspect the image evidence before deciding whether the corrected title is trustworthy.

Each candidate should prominently include, in this order:

- front image URL
- back image URL
- generated title
- corrected title
- evidence count
- feedback ids
- likely change types
- suggested decision options
- risk level
- install recommendation placeholder

For grouped patterns, review the representative evidence package:

- up to 5 representative front image URLs
- up to 5 representative back image URLs
- generated titles
- corrected titles
- feedback ids

The representative package exists so admins can visually inspect recurring mistakes, such as Gold Refractor vs Gold Wave, Cosmic Chrome vs Chrome, Patch vs Relic, or Auto vs Non-Auto.

Likely change types may include:

- `product`
- `set`
- `insert`
- `parallel`
- `serial`
- `player_subject`
- `auto_relic_patch`
- `grade`
- `wording_normalization`

Pattern types may include:

| Pattern Type | Meaning |
| --- | --- |
| `same_removed_phrase` | The same phrase was repeatedly removed from generated titles |
| `same_added_phrase` | The same phrase was repeatedly added to corrected titles |
| `same_replacement_phrase` | The same generated phrase was repeatedly replaced with the same corrected phrase |
| `repeated_corrected_phrase` | Multiple rows produced the same corrected title |
| `single_correction` | A correction did not create a stronger grouped pattern |

Risk levels are review hints only:

| Risk | Meaning |
| --- | --- |
| `low` | Usually formatting or narrow wording normalization |
| `medium` | Potentially useful rule with limited blast radius |
| `high` | Ambiguous, visual, broad, or low-evidence change |

High-risk candidates should usually become `Needs more evidence` unless the image evidence and checklist knowledge are strong.

## 5. Turn Admin Decisions Into Future Upgrade Proposals

Admins should review the Markdown file first, then inspect the JSON when exact fields are needed.

For each candidate, choose one decision:

- Accept as registry rule
- Accept as resolver rule
- Accept as prompt rule
- Accept as test case
- Ignore
- Needs more evidence

Use `Accept as test case` when the correction is important, image-specific, and should become a permanent regression example, but does not justify registry, resolver, or prompt changes.

Accepted candidates should become explicit upgrade proposals before any implementation work starts.

Each future upgrade proposal should include:

- evidence package with representative image URLs, generated titles, corrected titles, and feedback ids
- evidence count
- proposed change
- risk level
- affected files
- recommended tests
- admin notes

The evidence package should appear before proposed change, risk, and affected files in the proposal.

Only after a proposal is approved should a separate implementation step update:

- registry data
- resolver rules
- prompt rules
- test cases
- documentation

The offline review pipeline stops at candidate generation. It does not install upgrades.
