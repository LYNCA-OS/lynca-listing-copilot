## Purpose

Review Cycle #001 is the first offline admin review of Listing Copilot V2.0B feedback data.

The goal is to analyze the current Supabase feedback dataset and produce the first actionable upgrade report. This cycle stops at review and proposal preparation. It does not install upgrades.

## Hard Boundaries

This cycle must not:

- modify runtime title generation logic
- modify the registry
- modify resolver logic
- modify prompts
- deploy upgrades
- auto-approve candidates
- mutate raw feedback
- create RAG indexes
- fine-tune models
- require operator labels

All outputs are evidence for admin review only.

## Cycle Workflow

```text
Export Current Supabase Feedback
  |
Place Export In data/learning
  |
Run scripts/v2-learning-review.mjs
  |
Review JSON + Markdown Candidates
  |
Rank Upgrade Candidates
  |
Prepare Review Deliverables
  |
Admin Approval Checklist
```

## 1. Export Current Supabase Feedback Data

Export the current feedback rows from the V2.0B Supabase feedback table.

The export must include:

| Field | Required | Notes |
| --- | --- | --- |
| `id` | Yes | Stable feedback event id |
| `generated_title` | Yes | Title originally generated by Listing Copilot |
| `corrected_title` | Yes | Final title saved by the operator |
| `front_image_url` | Yes | Stable front image evidence URL |
| `back_image_url` | No | Stable back image evidence URL, when available |
| `operator_id` | Yes | Operator who saved the correction |
| `created_at` | Yes | Save timestamp |

Recommended export format:

```text
CSV or JSON
```

Recommended export filename:

```text
data/learning/supabase-feedback-export-cycle-001.csv
```

or:

```text
data/learning/supabase-feedback-export-cycle-001.json
```

Export rules:

- Export the full current feedback dataset for Cycle #001 unless a date range is explicitly chosen.
- Do not edit title text before processing.
- Do not remove rows because they look messy.
- Preserve image URLs exactly as exported.
- Preserve operator ids and timestamps exactly as exported.
- Keep rows with missing `back_image_url`; single-image records are valid.
- Mark rows with missing `front_image_url` as weaker evidence during review rather than deleting them.

## 2. Run The Offline Review Script

From the repository root, run:

```bash
node scripts/v2-learning-review.mjs data/learning/supabase-feedback-export-cycle-001.csv --date 2026-06-22
```

For JSON input:

```bash
node scripts/v2-learning-review.mjs data/learning/supabase-feedback-export-cycle-001.json --date 2026-06-22
```

The script writes:

```text
data/learning/review-candidates-2026-06-22.json
data/learning/review-candidates-2026-06-22.md
```

If Cycle #001 is run on a different calendar date, use that date in `YYYY-MM-DD` format.

The generated files are review artifacts only. They are not approved system changes.

## 3. Output Review Sections

The Cycle #001 upgrade report should summarize the generated review candidates using these sections.

All review sections should be evidence-first. For each candidate or grouped pattern, show representative image URLs before generated and corrected titles.

### Dataset Summary

Include:

- export filename
- export date
- review script command
- total input rows
- changed feedback rows
- skipped unchanged rows
- total review candidates
- date range of feedback rows
- number of operators represented
- number of rows with front image URLs
- number of rows with back image URLs

### Top Repeated Replacements

Summarize candidates where `pattern_type` is:

```text
same_replacement_phrase
```

Include:

- replacement phrase
- evidence count
- representative front image URLs
- representative back image URLs
- likely change types
- risk level
- example generated title
- example corrected title

### Top Added Phrases

Summarize candidates where `pattern_type` is:

```text
same_added_phrase
```

Use this section to find repeated missing title elements, such as serial numbers, parallels, inserts, grade wording, or auto/relic/patch terms.

### Top Removed Phrases

Summarize candidates where `pattern_type` is:

```text
same_removed_phrase
```

Use this section to find repeated hallucinated or over-applied title elements.

### Top Product Confusions

Summarize candidates where `likely_change_types` includes:

```text
product
```

Use this section to identify brand or product-line mistakes, such as confusing Prizm, Select, Optic, Chrome, Donruss, Bowman, or other product families.

### Top Insert Confusions

Summarize candidates where `likely_change_types` includes:

```text
insert
```

Use this section to identify missing, incorrect, or over-applied insert names.

### Top Parallel Confusions

Summarize candidates where `likely_change_types` includes:

```text
parallel
```

Use this section to identify visual distinction problems, such as Refractor vs Wave Refractor, Raywave, Mojo, Geometric, Sparkle, Shimmer, color parallels, or other surface treatments.

### Top Visual Confusions

Summarize image-specific candidates that require visual inspection before a rule can be trusted.

Include candidates where the correction depends on visible card evidence rather than wording cleanup alone.

Examples:

- Gold Refractor vs Gold Wave
- Cosmic Chrome vs Chrome
- Patch vs Relic
- Auto vs Non-Auto

For each visual confusion, include:

- up to 5 representative front image URLs
- up to 5 representative back image URLs
- generated titles
- corrected titles
- feedback ids
- evidence count
- likely change types
- risk level
- recommended decision option

Top Visual Confusions should be reviewed separately from wording changes. A visual candidate may become a test case even when it does not justify a registry or resolver change.

### Top Serial Corrections

Summarize candidates where `likely_change_types` includes:

```text
serial
```

Use this section to identify missing, incorrect, or misplaced serial numbering.

### Top Grade Corrections

Summarize candidates where `likely_change_types` includes:

```text
grade
```

Use this section to identify PSA, BGS, CGC, SGC, grade-number, qualifier, or slab wording corrections.

### Top Auto/Relic/Patch Corrections

Summarize candidates where `likely_change_types` includes:

```text
auto_relic_patch
```

Use this section to identify missing or incorrect autograph, relic, jersey, patch, RPA, logoman, memorabilia, or signature wording.

## 4. Upgrade Candidate Ranking

Rank candidates after human review of title pairs and image evidence.

### Priority 1: High Evidence + Low Risk

Use Priority 1 for candidates that have:

- repeated evidence across multiple feedback rows
- clear generated-to-corrected pattern
- low ambiguity
- low blast radius
- image evidence that supports the correction
- an obvious install path

Typical examples:

- repeated wording normalization
- repeated title-order correction
- repeated safe prompt-formatting correction
- clear registry addition with narrow scope and multiple evidence rows

Priority 1 candidates are the best first upgrade proposals.

### Priority 2: High Evidence + Medium Risk

Use Priority 2 for candidates that have:

- repeated evidence
- meaningful system value
- some ambiguity or broader blast radius
- need for targeted tests before installation

Typical examples:

- resolver rule with limited scope
- product-family correction that may affect neighboring sets
- insert or parallel correction supported by several examples
- prompt extraction-priority change

Priority 2 candidates may become upgrade proposals after stronger review notes and recommended tests are attached.

### Priority 3: Needs More Evidence

Use Priority 3 for candidates that have:

- low evidence count
- unclear images
- subtle visual distinction
- possible operator preference
- broad potential impact
- uncertain checklist or registry knowledge

Typical examples:

- single-row visual parallel correction
- player or subject change without clear image evidence
- broad resolver idea inferred from limited data
- high-risk registry mutation

Priority 3 candidates should not be installed. They should be tracked for future cycles.

## 5. Review Deliverables

Cycle #001 should produce four deliverable groups.

### Registry Candidates

Registry candidates are corrections that may become durable card knowledge.

Include:

- product corrections
- set corrections
- insert corrections
- parallel corrections
- player or subject corrections
- serial or memorabilia naming conventions
- grade canonicalization when it belongs in durable knowledge

Each registry candidate must include evidence count, examples, image URLs, risk level, and recommended tests.

### Resolver Candidates

Resolver candidates are corrections that may become deterministic decision logic.

Include:

- image evidence priority rules
- front/back evidence conflict handling
- visual distinction rules
- serial extraction behavior
- grade/slab interpretation behavior
- auto/relic/patch interpretation behavior

Each resolver candidate must include positive examples, negative examples to collect or create, and blast-radius notes.

### Prompt Candidates

Prompt candidates are corrections that may become instruction or output-format changes.

Include:

- title ordering corrections
- wording normalization
- missing visible serial instruction
- missing visible grade instruction
- unsupported assumption avoidance
- exact visible insert-name preservation

Prompt candidates should not encode large factual registries.

### Test Case Candidates

Test case candidates are examples that should become regression coverage after approval.

Include:

- front/back image URLs
- generated and corrected title pair
- likely change type
- expected future behavior
- negative behavior to prevent
- source feedback ids

Use test case candidates when:

- the correction is important
- the correction is image-specific
- the case should become a permanent regression example
- the correction does not justify registry or resolver changes

Test cases may be approved even when no registry, resolver, or prompt change is approved yet.

## 6. Approval Checklist Before Installation

No upgrade may be installed from Cycle #001 until this checklist is complete.

Approval checklist:

- The candidate has been reviewed by an admin.
- The original `generated_title` and `corrected_title` were inspected.
- Front/back image evidence was inspected when available.
- The corrected title is believed to be right, not merely different.
- The likely change type is confirmed or corrected by the admin.
- The candidate has been assigned Registry, Resolver, Prompt, Test Case, Documentation, Ignore, or Needs More Evidence status.
- `Accept as Test Case` was considered for image-specific corrections that should become regression examples.
- Evidence count is recorded.
- Risk level is recorded.
- Affected files are listed.
- Recommended tests are listed.
- Blast radius is considered.
- High-risk candidates have additional evidence or are deferred.
- The upgrade proposal is written separately from raw review output.
- Approval is recorded before implementation starts.

Installation remains out of scope for this document.

Review Cycle #001 is complete when:

- current Supabase feedback has been exported
